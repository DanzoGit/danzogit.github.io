# Анализ изменения `ignore_loot_abundance` (7DTD 2.6 b14 → 3.0 b252)

## TL;DR

В 3.0 разработчики убрали атрибут `ignore_loot_abundance="true"` со **всех мировых
контейнеров** (сейфы, сундуки, ящики, почтовые ящики, закопанные припасы, награды
за квесты и т.д.) и начали переход на новую покатегорийную систему изобилия через
атрибут `abundance_type` на `lootgroup`.

Это **не случайная поломка** — направление изменения намеренное. Но в b252 миграция
**не завершена**, и это даёт ощутимый побочный эффект:

- При настройке `LootAbundance = 100%` (по умолчанию) количество лута не меняется.
- При `LootAbundance < 100%` (например, 25–50%) ранее «иммунные» контейнеры теперь
  **тоже урезаются** глобальным ползунком. Раньше сейфы/сундуки/тайники выдавали
  полный набор независимо от ползунка — теперь нет. Игроки с пониженным изобилием
  получают заметно меньше лута, чем в 2.6.
- Новая покатегорийная система (`abundance_type`) пока **не подключена** к настройкам:
  множители категорий захардкожены в 1.0, отдельных ползунков в игре нет.

Итог: единственный действующий регулятор количества — глобальный `LootAbundance`,
и теперь он бьёт по тем контейнерам, которые в 2.6 были от него защищены.

## Что именно изменилось в данных

Сравнение `Config/loot.xml`:

| Показатель | 2.6 b14 | 3.0 b252 |
|---|---|---|
| Всего `lootcontainer` | 337 | 339 |
| С `ignore_loot_abundance="true"` | 184 | 138 |
| └ из них twitch-контейнеры | 138 | 138 |
| └ из них **мировые** контейнеры | **46** | **0** |
| `lootgroup` с атрибутом `abundance_type` | 0 | 68 |

Иначе говоря: **все ~46 мировых контейнеров потеряли `ignore_loot_abundance`**,
а атрибут сохранился исключительно у `twitch_*` контейнеров (награды за
стримерские события).

### Новый атрибут `abundance_type`

В 3.0 у групп лута появился `abundance_type` (комментарий в самом loot.xml:
`Loot group parameter abundance_type is for loot abundance Sandbox Options`).
Возможные значения (из enum `LootContainer.LootGroup.AbundanceLootModTypes`):
`Food, Drinks, Ammo, Meds, Resources, Armor, Melee, Ranged, Dukes, Magazines`.

То есть концепция «изобилия» задумана с переездом с уровня **контейнера** (бинарный
opt-out) на уровень **категории предметов** (множитель на тип лута).

## Как это работает в коде (декомпил Assembly-CSharp 3.0 b252)

### Парсинг
- `LootFromXml.cs`: всё ещё читает `ignore_loot_abundance` → поле
  `LootContainer.ignoreLootAbundance`.
- `LootFromXml.cs:373` — новый разбор: `if (_elementGroup.HasAttribute("abundance_type"))`
  → `Enum.TryParse(..., out lootGroup.abundanceType)`.

### Множители (`LootContainer.cs`)
```csharp
public static float GlobalCountModifier = 1f;   // глобальный (LootAbundance)
public static float FoodCountModifier   = 1f;   // покатегорийные
public static float ArmorCountModifier  = 1f;
public static float MagazinesCountModifier = 1f;
// ... и т.д. (Drinks, Ammo, Meds, Resources, Melee, Ranged, Dukes)
```

- `GlobalCountModifier` задаётся из `EnumGameStats.LootAbundance`
  (`GameStatsBridge.cs:70: LootContainer.GlobalCountModifier = ToFloatPercent(_newValue)`).
- Покатегорийные модификаторы (`FoodCountModifier` и др.) в коде **только читаются**
  (`GetCountMultiplierFromSandbox`, `GetCountFromSandbox`), но **нигде не присваиваются** —
  остаются `1f`. В `EnumGameStats` есть только глобальный `LootAbundance`, а в меню
  настроек (`XUi_Menu/windows.xml`) — единственный ползунок лута
  `LootAbundance` со значениями `25, 33, 50, 67, 75, 100, 150, 200`.
  → Покатегорийная часть в b252 **ещё не подключена** к интерфейсу и всегда даёт 1.0.

### Логика спавна — `LootContainer.Spawn(...)`
```csharp
float abundance = 1f;
if (ignoreLootAbundance) {                 // контейнер с ignore_loot_abundance="true"
    num = RandomSpawnCount(min, max, 1f);  // полный объём, ползунок игнорируется
    abundance = 1f;
}
else if (GlobalCountModifier > 1f) {       // LootAbundance > 100%
    ... abundance = GlobalCountModifier;   // больше лута
}
else {                                     // LootAbundance <= 100% (вкл. 100%)
    num = RandomSpawnCount(min, max, 1f);
    abundance = GlobalCountModifier;       // при 100% == 1.0; при <100% — меньше
}
```
`RandomSpawnCount(...)` в конце делает `num *= abundance`, то есть `abundance`
прямо масштабирует число выпадающих предметов. Поскольку мировые контейнеры
больше не попадают в первую ветку (`ignoreLootAbundance`), они идут в общую и
теперь множатся на `GlobalCountModifier`.

## Поведение: было / стало

| LootAbundance | Мировой контейнер 2.6 (с ignore) | Мировой контейнер 3.0 (без ignore) | Twitch 2.6/3.0 (с ignore) |
|---|---|---|---|
| 100% (по умолчанию) | полный | **полный (без изменений)** | полный |
| 75% | полный (иммунитет) | **~75%** | полный |
| 50% | полный (иммунитет) | **~половина** | полный |
| 25% | полный (иммунитет) | **~четверть** | полный |
| 150% | полный (иммунитет) | **больше** | полный |

Ключевой момент: на 100% лут идентичен 2.6; при любом понижении ползунка мировые
контейнеры теперь режутся, чего в 2.6 не было.

## Случайность или умысел?

### За умышленность (направление изменения)
1. Изменение **системное и консистентное**: атрибут убран сразу со всех 46 мировых
   контейнеров, без единого исключения — не похоже на случайную потерю строк.
2. Одновременно введён **новый механизм** `abundance_type` (68 групп) с явным
   комментарием про «loot abundance Sandbox Options». Старый per-container opt-out
   конфликтует с покатегорийным управлением: если бы сейф игнорировал изобилие,
   покатегорийные ползунки не влияли бы на его содержимое.
3. Атрибут **намеренно сохранён только у `twitch_*`** контейнеров: награды за
   стримерские события обязаны выдавать ровно заданный набор при любых настройках.

### За «недоделку» (почему это ощущается как поломка)
1. Покатегорийные множители (`FoodCountModifier`, `ArmorCountModifier`, …) объявлены
   и читаются, но **не привязаны ни к одной настройке** — всегда 1.0; в меню есть
   только глобальный ползунок.
2. Пример группы с `abundance_type` в шапке loot.xml оставлен **закомментированным**
   (`<!-- <lootgroup name="skillMagazines" abundance_type="Magazines"> -->`).
3. Защиту (`ignore_loot_abundance`) сняли **до того**, как новая система заработала.
   В итоге на пониженном изобилии игрок теряет лут из «гарантированных» мест —
   а замены (покатегорийной точной настройки) ещё нет.

→ Это **поэтапный рефакторинг, выпущенный в промежуточном состоянии**: направление
задумано осознанно, но в b252 наблюдается реальная просадка лута при `LootAbundance < 100%`.

## Практические последствия

- **На 100% изобилия** — изменений нет.
- **На пониженном изобилии** (типичный выбор для усложнения) сейфы, сундуки,
  спец-ящики, закопанные припасы, награды за квесты, сумки с добычей с зомби и пр.
  теперь выдают **меньше**, тогда как в 2.6 они были защищены. На 25% это ~четверть
  от прежнего объёма из этих мест.
- **На повышенном изобилии** те же контейнеры дают **больше**.
- **Twitch-награды** — без изменений (фиксированы при любых настройках).

## Вывод для обычных игроков

Если у тебя стоят обычные настройки (количество добычи — 100%), то ничего не
изменилось: добычи столько же, сколько и раньше.

А вот если ты понизил настройку количества добычи (например, поставил 25% или 50%
ради сложности), то теперь добычи стало заметно меньше, чем в прошлой версии игры.

Дело вот в чём. Раньше «особые» места всегда отдавали полное содержимое и не
обращали внимания на этот ползунок:

- сейфы и оружейные сейфы;
- прочные, деревянные, кожаные и укрепленные сундуки;
- замаскированные тайники, закопанные припасы и тайники с сокровищами;
- зараженные тайники из заданий;
- почтовые ящики, книжные шкафы, банкоматы;
- фирменные коробки и ящики из магазинов и зданий;
- сумки с оружием;
- сброс груза с самолета;
- награды за задания;
- сумки с добычей, которые роняют зомби.

Теперь все эти места тоже слушаются ползунка. Поставил 25% — получишь примерно
четверть прежнего и из них. Чем ниже ползунок, тем сильнее просадка, и бьёт она
как раз по ценным «гарантированным» местам, ради которых обычно сложность и
повышают.

Что с этим делать:

- Хочешь как раньше — верни количество добычи на 100%.
- Играешь на пониженной добыче ради сложности — просто знай, что теперь сейфы и
  сундуки тоже дают меньше. Это не ошибка твоего сохранения и не баг конкретно у
  тебя — так теперь устроена игра.

Единственное исключение — награды за события на трансляциях (для стримеров): они
по-прежнему выдают ровно столько, сколько задумано, при любых настройках.
